## Create an AMF configuration
To use AMF you need an `AMFConfiguration` object.
This immutable object stores and configures:
- Parsing options
- Render options
- Resource loaders
- Plugins
- Transformation pipelines
- Validation profiles
- Error handler providers
- Event Listeners
- And more ...

## Predefined configurations
AMF has predefined configurations for each specification it supports:
- `RAMLConfiguration`
    - For RAML 0.8 and RAML 1.0 documents
- `OASConfiguration`
    - For OAS 2.0 and OAS 3.0 documents
- `AsyncAPIConfiguration`
    - For AsyncAPI 2.0.0 documents

Each configuration has all it needs to parse, transform, validate and render documents.

Moreover, AMF has general-purpose configurations that support spec-agnostic parsing.
This enables parsing without knowing the exact specification of a document.
These configurations are:
- `WebAPIConfiguration`
  - Can parse both RAML and OAS documents
- `APIConfiguration`
  - Can parse RAML, OAS, and AsyncAPI documents

The `AMFResult` returned by parsing has the exact specification parsed.
All other operations (transformation, validation, rendering) are spec-dependent and require a specific configuration.

## Create a client and use AMF
To interact with AMF you need to create a client from a configuration.
There are 2 types of clients in AMF, depending on what you want to work with:
- `baseUnitClient`
    - Contains common AMF operations to operate with `BaseUnit`
    - A `BaseUnit` is the model used by AMF to store parsed documents
- `elementClient`
    - Contains functionality associated with specific elements of the AMF model
